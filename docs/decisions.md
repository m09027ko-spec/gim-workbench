# 設計判断ログ（Decisions）

このファイルは「なぜそうしたか」の根拠を残します。後から「なぜこの選択？」と疑問が湧く判断だけを書きます。手順書や現状スナップショットは対象外です。

形式: 各判断は `## YYYY-MM-DD: 一行タイトル` で始め、**背景／決定／代替案／結果** の4項目で書きます。判断が覆ったときは、新しい日付で「上書き」エントリを追加してください（過去エントリは削除しない）。

---

## 2026-05-17: ルーティングは HashRouter

- **背景**: GitHub Pages は静的ホスティングのため、`/category/ward` のような BrowserRouter のパスはリロード時に 404 になる。
- **決定**: HashRouter（`#/category/ward`）を採用。
- **代替案**: BrowserRouter + 404 リダイレクトハックも可能だが、PWA との相性が悪く、サブパス公開時の事故が多い。
- **結果**: スマホSafariのリロード・ホーム画面追加で安定動作。GitHub Pages 上でリンク共有しても壊れない。

---

## 2026-05-17: 外部API・AI API を使わない

- **背景**: 患者情報や症例本文を扱う可能性のあるアプリで、サーバー送信経路があると個人情報流出のリスクが構造的に残る。
- **決定**: 外部通信を一切持たない。CDN からのライブラリ読み込みもしない。
- **代替案**: 自前APIを立てて匿名化処理する案もあるが、運用負担と事故リスクが過大。
- **結果**: GitHub Pages 上で静的 HTML/JS/CSS のみで完結。アプリ自体が「送信しない」設計なので、利用者側の運用注意点が最小化される。

---

## 2026-05-17: UIライブラリを導入しない

- **背景**: MUI / Chakra / Tailwind を入れるとビルドサイズと依存更新負担が増える。ユーザー（医師）はプログラミング初心者で、依存を最小に保ちたい。
- **決定**: 通常CSS + 自作コンポーネントで作る。
- **代替案**: Tailwind は CSS のみで済むので候補だったが、クラス名の習熟コストが医師にとって障壁。
- **結果**: `src/styles/global.css` 一本で全画面を統制。依存が React / React Router / Vite だけで済む。

## 2026-05-17: 文書フォームの自由記載を localStorage に保存しない

- **背景**: 文書テンプレートの自由記載欄に、つい患者氏名・症例本文を入れてしまう運用リスクがある。一度ローカル保存すると、端末紛失や共有時に事故になる。
- **決定**: 文書フォームの入力値は React の state のみで保持し、画面遷移・リロードで消える設計。
- **代替案**: ローカル暗号化保存も検討したが、複雑化に見合わない。
- **結果**: 入力 → コピー → カルテに貼り付け、で完結。データはアプリに残らない。

## 2026-05-17: モジュールは3つの型に固定

- **背景**: モジュールが 30 個近く増える見込みで、毎回独自実装すると保守不能。
- **決定**: チェックリスト型・計算型・文書型の3型に必ず当てはめる。新しい型は議論なしに増やさない。
- **代替案**: フレームワーク化（プラグイン方式）も検討したが、現時点では過剰設計。
- **結果**: 共通部品（`Checklist` / `scoring.ts` / `textTemplates.ts`）で型を吸収。新規モジュール追加のコストが下がり、エージェントも「どの型か」を仕様書から読むだけで実装パターンが決まる。

## 2026-05-17: モジュールIDで横串

- **背景**: 仕様書（`docs/specs/`）・カタログ定義（`src/data/catalog/`）・実装（`src/modules/`）が別ディレクトリに散らばっており、対応関係が見えにくい。
- **決定**: 3点すべてを同じ `<module-id>` で揃える規約に。エージェントは ID を頼りに 3 ファイルを機械的に辿れる。
- **代替案**: モジュール単位フォルダ化（`src/modules/<category>/<module-id>/` にコンポーネントと付属ファイルを集約）も検討したが、現時点では 1 ファイルで足りるモジュールが大半なので過剰。
- **結果**: 仕様書ファイル名・カタログのID・コンポーネント名が一意に対応する規約に。将来モジュール単位の付属ファイルが増えたら、そのモジュールだけフォルダに昇格する漸進的拡張に倒す。

## 2026-05-17: 成果物だけ Git に上げる（参考資料はローカル）

- **背景**: 論文 PDF・ガイドライン本文をリポジトリに入れると、著作権リスクと「うっかり患者情報を含む資料が混入する」リスクが両方発生する。
- **決定**: `docs/references/` は `.gitignore` で除外し、README のみ Git に上げる。資料は先生のローカルにだけ存在し、Cowork が要約と引用情報を `docs/specs/<module-id>.md` に書き写すことで、成果物として Git に残る。
- **代替案**: Private リポジトリにすれば資料そのものを上げられるが、GitHub Pages 公開と両立しにくく、運用が複雑。
- **結果**: リポジトリは「世界に見せていい成果物だけ」という線が明確に。参考資料の収集と整理は先生のローカルで自由にでき、引用は仕様書経由で記録される。

## 2026-05-17: AGENTS.md と CLAUDE.md を分離

- **背景**: 全エージェント共通の不変則と Claude 固有のメモを1ファイルにまとめると、Codex / Cursor を将来併用するとき二重管理が発生する。
- **決定**: 共通部分は `AGENTS.md` に集約し、`CLAUDE.md` は冒頭で AGENTS.md を参照させ、Claude 固有のメモだけ残す。
- **代替案**: 1ファイル運用も可能だが、将来の保守コストが上がる。
- **結果**: Codex や別エージェントを試すとき、`AGENTS.md` を読ませるだけで同じプロジェクト規約が効く。Claude 固有の運用上の癖（TodoWrite の刻み方、報告スタイル）は CLAUDE.md に閉じる。

## 2026-05-17: 30ツールは定義駆動の共通レンダラーへ集約

- **背景**: 総診ワークベンチ内に30個の内科ミニツールを一気に実装する依頼に変わり、各ツールを個別コンポーネント化すると重複が大きくなる。
- **決定**: `src/data/tools.ts` にツール定義、入力項目、結果ロジックを集約し、`src/modules/miniTools/MiniToolRenderer.tsx` で共通表示する。計算式は `src/utils/calculations.ts`、薬剤・チェックリストデータは `src/data/*.ts` に分離する。
- **代替案**: 30ツールをそれぞれ `src/modules/<category>/` の個別コンポーネントにする案もあったが、共通免責、入力UI、結果カード、TODO表示が重複する。
- **結果**: 30ツールを同じページ構造で実装でき、薬剤量など未確認データはTODOとして安全に表示できる。
