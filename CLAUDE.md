## Claude Code 固有のメモ

### TodoWrite の使い方

非自明な作業は TodoWrite でタスクに刻む。1モジュール実装あたり 5〜8 タスクが目安。

例（新規モジュール実装の場合）:

1. 仕様書を読む
2. カタログ定義を追加
3. コンポーネント実装
4. `src/data/tools.ts` に登録
5. `npm run build` を通す
6. `docs/PROJECT_MAP.md` を更新
7. `docs/HANDOFF.md` を更新

### ビルド完了の定義

`npm run build` が成功するまでが完了。型エラー・lint警告を握りつぶさず、原因を報告してください。

### ユーザーへの報告スタイル

- ユーザーは医師でプログラミング初心者。専門用語は避けるか一行で補足。
- 「何を変えたか」「なぜそうしたか」を簡潔に。
- 長い後語り、diff の再要約は不要。
- コードを長々と貼らない。変更点のサマリで十分。

### 仕様書がない依頼を受けたとき

ユーザーが詳細仕様を依頼文に含めている場合は、その依頼文を仕様として扱って実装してよいです。仕様が曖昧で医学的判断や施設ルールが必要な場合は、未確認値をTODO表示にして安全に進めます。

`docs/specs/<module-id>.md` も詳細依頼文もない新規モジュール依頼の場合は、実装に進まず以下を返してください。

> このモジュールの仕様書（`docs/specs/<module-id>.md`）がまだ無いようです。Cowork（Claude Desktop の Cowork mode）で仕様書を作ってから、改めて実装を依頼してください。

### 絶対則違反になりそうな依頼を受けたとき

`AGENTS.md` の絶対則に抵触する依頼は、実装に進まず根拠を添えてユーザーに確認してください。「外部APIを使ってください」「localStorage に症例文を保存してください」などが該当します。

### Git操作

実装完了後、ユーザーから明示的に依頼があれば `git add` → `git commit` → `git push` まで一気通貫で行ってよい。コミットメッセージは `AGENTS.md` の「コミットメッセージとログの規約」セクションに従う。Claude Code が手を動かした場合のエージェント識別子は `[claude-code]`。

例: `feat(has-bled): implement HAS-BLED calculator [claude-code]`
